9단계

Tabelog 별점·후기 수를 포함한 식당 alt 36개 추가, 공유 rating 스키마와 alt 헤더·메인 맛집 카드 인라인 표시 구현.

Restaurant Ratings

왜

Stage 8에서 alt 후보를 지도에 표시했지만 식당이 적었고, 무엇보다 "이 가게가 진짜 좋은가"를 판단할 신뢰 지표가 없었다. 폰으로 후보를 둘러보면서 "Tabelog 3.8, 후기 2,000개" 같은 정보가 인라인으로 보이면 방문 결정이 훨씬 빠르다.

무엇을

  • 스키마: 모든 place에 옵셔널 place.rating / place.review_count / place.rating_source; place.restaurants[]에도 동일 3개 필드; validator에 공유 validateRating 헬퍼
  • UI: renderRating(obj) 헬퍼가 "★ 4.3 · 후기 1,234개 (Tabelog)" 포맷으로 인라인 표시(있는 필드만); amber-600 색상, 소형 폰트
  • alt 장소 디테일 헤더의 name_jp 아래, 메인 명소 맛집 카드 type 라인 옆에 표시
  • 데이터: Day1~4 area별 식당 alt 36개 신규(라멘·오코노미야키·타코야키·스시·야끼니쿠·쿠시카츠·우동·소바·카츠·디저트·이자카야·카레), 대부분 Tabelog 별점 포함; 메인 명소 맛집 카드 1개 sample 별점 추가로 양쪽 UI 경로 모두 검증

어떻게

스키마+샘플(1건)로 인라인 표시를 먼저 확인한 뒤, 4개 병렬 subagent가 Day1~4 식당 후보를 작성. 각 subagent는 /tmp/restaurant-patches/dayN-food.json에 결과 저장; 단일 merge 스크립트로 적용하면서 Day1·Day3 중복(ichiran_dotonbori 등)을 dedupe. http:// URL 1건 자동으로 https://로 업그레이드. Tabelog 점수에는 한국 사용자가 5점 만점 체계와 혼동하지 않도록 (Tabelog) 출처 표기를 함께 표시.

코드 위치

  • scripts/validate-data.mjs:20-37 (validateRating 헬퍼)
  • scripts/validate-data.mjs:115-128 (place 및 restaurants[]에 호출)
  • assets/app.js:37-46 (renderRating 헬퍼)
  • assets/app.js:48-67 (renderRestaurant rating 인라인)
  • assets/app.js:416 (renderDetail 헤더 rating)
  • data/places.json (36개 신규 alt 식당 + 1개 샘플 메인 맛집 별점)

회고

스키마 옵셔널화로 기존 95개 데이터셋에서 회귀가 없었다. 4개 병렬 subagent가 약 7분에 36개 식당 큐레이션을 완료했다. Tabelog 출처 표기는 의도된 UX 선택이다. 3.0~4.0 분포는 일본 사용자에게는 의미 있지만, 한국 사용자에게는 맥락 없이 "낮다"고 보일 수 있기 때문이다.
