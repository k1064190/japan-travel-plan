2단계

5일치 65개 장소 완전 데이터, TDD validator, 메인 명소별 맛집 및 활동 항목 큐레이션.

Data

왜

Stage 1 와이어링이 동작함을 확인했으니, 진짜 정보를 채워야 했다. 사용자가 현장에서 바로 따라갈 수 있는 수준 — 좌표가 정확하고, 출처가 있고, 메인 명소엔 맛집이 있어야 했다.

무엇을

  • data/places.json: 총 65개 명소 — Day1: 12, Day2: 21(교토), Day3: 26(오사카), Day4: 4(USJ), Day5: 2(출국)
  • data/itinerary.json: Day1~Day5, 21개 stop, 모든 stop에 transit_from_prev 및 duration_minutes
  • scripts/validate-data.mjs: 스키마 / 일본 좌표 범위 / place_id 참조 / URL 안전성 / main→restaurants(utility 태그 예외) 검증
  • tests/validate-data.test.mjs: 8개 테스트 케이스 통과
  • package.json: npm test, npm run validate 스크립트

어떻게

Day별 subagent를 병렬 dispatch. 각 subagent에 마스터 카탈로그 해당 부분과 JSON 스키마를 전달하고, 좌표는 WebSearch로 추출 후 도시별 경계 검증. 결과를 메인이 places.json에 머지. Validator는 TDD로 먼저 작성했으며, 첫 실행에서 발견된 정책 빈틈(utility 태그 예외)은 즉시 8번째 테스트로 영속화됐다.

코드 위치

  • data/places.json (65개 항목)
  • data/itinerary.json (5일, 21개 stop)
  • scripts/validate-data.mjs (validate(places, itinerary))
  • tests/validate-data.test.mjs (8개 테스트)
  • package.json (test, validate 스크립트)
  • docs/stage-2/place-catalog.md (마스터 리스트)

회고

Day별 분리 dispatch로 60+ 명소 조사를 약 25분 안에 완료했다. TDD validator가 첫 실행에서 정책 빈틈을 정확히 잡아내 즉시 테스트로 고정됐다. 검증된 맛집(奥丹清水 1635년, 串かつだるま 원조, ARABICA, 茶寮都路里 등)은 현장에서 찾기 쉽다.

Place Catalog

왜

병렬 subagent dispatch 전에 일관된 위치 ID, 한국어 명칭, main/alt 분류를 정리한 기준표가 필요했다.

무엇을

  • 5일치 64개 장소 카탈로그(Day1: 12, Day2: 21, Day3: 26, Day4: 4, Day5: 2)
  • 두 가지 분류: main(맛집 포함 전체 프로필), alt(요약·좌표·활동·후기 링크만)
  • 각 항목: ID, 한국어 명칭, 분류, 컨텍스트(예: "아침 거점", "숙소 후보")

어떻게

여행 계획 설계와 GPT 일정 입력을 기반으로 초안 작성. 이 카탈로그가 Stage 2 데이터 작성의 단일 기준이 되어 places.json과 itinerary.json 사이의 ID 일관성을 보장했다.

코드 위치

  • docs/stage-2/place-catalog.md (마스터 리스트, 64개 항목)

회고

카탈로그 우선 접근으로 병렬 subagent dispatch 중 ID 불일치를 방지했다. 최종 수량은 65개(Day2 alt 1개 추가)로 소폭 증가했으나 의도된 보완이다.
