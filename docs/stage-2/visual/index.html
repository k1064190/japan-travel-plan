Stage 2

Complete data for 65 places across all 5 days, with a TDD validator and curated restaurant/activity entries per main location.

Data

Why

With Stage 1's wiring confirmed, the real information needed to be filled in. The goal was field-usable data: accurate coordinates with sources, restaurants for main locations, and live review-search links.

What

  • data/places.json: 65 places total — Day 1: 12, Day 2: 21 (Kyoto), Day 3: 26 (Osaka), Day 4: 4 (USJ), Day 5: 2 (departure)
  • data/itinerary.json: Days 1–5, 21 stops, all with transit_from_prev and duration_minutes
  • scripts/validate-data.mjs: schema / Japan coordinate bounds / place_id references / URL safety / main → restaurants (utility tag exceptions) validation
  • tests/validate-data.test.mjs: 8 test cases passing
  • package.json: npm test and npm run validate scripts

How

One subagent per day dispatched in parallel. Each received the relevant portion of the master catalog plus a strict JSON schema. Coordinates were extracted via WebSearch from Wikipedia and official sites, then checked against city-level bounds. Results were merged into places.json by the main implementer. The validator was written TDD-first; a policy gap found during the first run (utility-tagged main places need no restaurants) became the 8th test case.

Code Locations

  • data/places.json (65 entries)
  • data/itinerary.json (5 days, 21 stops)
  • scripts/validate-data.mjs (validate(places, itinerary))
  • tests/validate-data.test.mjs (8 tests)
  • package.json (test, validate scripts)
  • docs/stage-2/place-catalog.md (master list)

Retrospective

Parallel day-by-day dispatch finished 60+ locations in roughly 25 minutes. The TDD validator caught a policy gap on the first run (utility-tag exception), which was immediately locked in as a test case. Verified restaurants — Okutan Kiyomizu (est. 1635), Kushikatsu Daruma original, ARABICA, Tsujiri — are easy to find in the field.

Place Catalog

Why

A structured master list was needed before dispatching parallel subagents, so every day had an authoritative set of location IDs, Korean names, and main/alt classifications to fill against.

What

  • 64 places catalogued across 5 days (Day 1: 12, Day 2: 21, Day 3: 26, Day 4: 4, Day 5: 2)
  • Two categories: main (full profile with restaurants) and alt (lite profile with summary, coordinates, activities, review links)
  • Each entry maps to an id, Korean name, category, and source context (e.g., “morning anchor”, “hotel candidate”)

How

Drafted from the travel plan design and GPT itinerary input. The catalog served as the single source of truth during Stage 2 data population, ensuring consistent IDs across places.json and itinerary.json.

Code Locations

  • docs/stage-2/place-catalog.md (master list, 64 entries)

Retrospective

The catalog-first approach prevented ID mismatches during parallel subagent dispatch. The final count reached 65 (one Day 2 alt added during population), a minor and intentional adjustment.
