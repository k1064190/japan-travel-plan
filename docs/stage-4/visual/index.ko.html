4단계

21개 stop 전체, 해시 라우팅 엣지 케이스, iOS safe-area inset, 크로스브라우저 검증, README 작성을 포함한 QA 통과.

QA

왜

마지막 가드레일: (1) 21개 stop이 전부 클릭으로 열리는가, (2) URL 공유와 잘못된 hash가 graceful하게 처리되는가, (3) iOS 모바일 안전영역에 탭바가 가려지지 않는가, (4) README만으로 앱을 실행할 수 있는가.

무엇을

  • scripts/smoke-test.md — 21개 stop 회귀 체크리스트(Day별 place_id 표 + 라우팅/모바일/크로스브라우저 항목)
  • assets/app.js — hash 라우팅 정규화 + 디테일 패널 자동 닫기 로직 + buildHash() 헬퍼(리뷰 피드백 반영 리팩토링)
  • assets/styles.css — #mobile-tabs에 padding-bottom: env(safe-area-inset-bottom, 0) 추가
  • README.md — 실행/데이터 수정/검증/디렉토리 구조 안내
  • 최종 리뷰 트리오 통과(code-reviewer-pro + gemini-subagent; Codex는 MCP 도구 schema 오류로 fallback)

어떻게

헤드리스 Chromium 스크립트로 21개 stop을 hash 변경으로 자동 순회해 디테일 헤더 로드를 확인. 두 라우팅 엣지 케이스 발견 및 수정: 잘못된 hash(#day9/foo)가 stale UI를 남기는 문제를 history.replaceState 정규화와 디테일 패널 자동 숨김으로 해결. 같은 날 stop 전환과 모바일 뷰포트(390×844)도 검증. 최종: node scripts/validate-data.mjs → 65개 장소, 5일; node --test tests/validate-data.test.mjs → 11개 통과.

코드 위치

  • scripts/smoke-test.md
  • README.md
  • assets/app.js:325-336 (buildHash 헬퍼)
  • assets/app.js:344-355 (boot() URL 정규화 + optional chaining)
  • assets/app.js:365-400 (hashchange 핸들러)
  • assets/styles.css:81-85 (#mobile-tabs safe-area-inset)

회고

헤드리스 smoke 스크립트가 수동 리뷰에서는 잡지 못했던 두 라우팅 엣지 케이스를 잡아냈다. Codex PR 리뷰(3라운드, 16건)는 모두 기계적인 방어적 수정이라 명확성 없이 전부 적용됐다. parseHash의 silent fallback 패턴은 향후 SPA에서 valid: false 시그널 방식으로 개선할 여지가 있다.
