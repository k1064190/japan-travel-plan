5단계

65개 전체 명소에 195개 한국어 블로그 큐레이션 링크 추가. 맛집·활동별 인라인 링크, 일반 검색 버튼을 직접 게시글 진입으로 대체.

Curated Content

왜

기존 "한국어 후기 검색"은 네이버·구글·유튜브 검색창 진입 링크 3개에 불과해 현장에서 매번 직접 검색·정렬·평가하는 시간이 들었다. 인터넷에 신뢰할 만한 한국어 후기가 충분하므로, 미리 큐레이션해 데이터에 박아두면 클릭 한 번에 본문으로 바로 진입할 수 있다.

무엇을

  • 스키마 확장: 명소 레벨 curated_links: [{title, url, source, snippet?}], 맛집 레벨 link: {title, url, source}, 활동 항목 string | {text, link?}(역호환)
  • Validator 강화: 모든 URL https:// 강제, 비어있지 않은 title/source 필수
  • UI 헬퍼 3개 신규: renderActivity, renderRestaurant, renderCuratedLink
  • 큐레이션 블로그 카드 그리드(제목+출처+1줄 스니펫), 활동·맛집 인라인 링크, 기존 검색 그리드는 "더 찾아보기" 폴백으로 격하
  • 65/65 명소: 각 3개 큐레이션 링크(총 195개), 40/41 메인 맛집: 각 1개 링크, 48개 메인 활동: 객체 형태로 래핑

어떻게

Phase A: 스키마+UI+Day1 메인 3개로 패턴 검증. Phase B: 7개 background subagent를 day/분류별로 병렬 dispatch(Day2 main, Day3 main, Day4~5 main, Day1 alt, Day2 alt, Day3 alt, Day4 alt). 각 subagent가 WebSearch로 한국어 블로그 검색 → 신뢰 가능한 글 큐레이션 → /tmp/curated-patches/에 patch JSON 저장. 단일 merge 스크립트가 7개 patch를 places.json에 일괄 적용(충돌 없음, subagent는 read-only).

코드 위치

  • scripts/validate-data.mjs:20-39 (validateCuratedLink 헬퍼)
  • scripts/validate-data.mjs:84-115 (curated_links / activities / restaurants 옵셔널 검증)
  • assets/app.js:28-66 (renderActivity, renderRestaurant, renderCuratedLink)
  • assets/app.js:222-265 (renderDetail 큐레이션 섹션 + 폴백 검색 그리드)
  • data/places.json (65개 명소 전체 curated_links 추가)

회고

7개 병렬 subagent로 62개 명소 큐레이션을 wall-clock 약 6분에 완료했다. patch JSON + 단일 라이터 merge 패턴으로 충돌을 원천 차단했다. 스키마 옵셔널화로 65개 전체 데이터셋에서 회귀 0. "더 찾아보기" 폴백 그리드를 유지해 큐레이션이 부족할 때도 즉시 추가 검색이 가능하다.
