Stage 5

Curated Korean Blog Links

Why

기존 "한국어 후기 검색"은 네이버·구글·유튜브 검색창 진입 링크 3개에 불과해 현장에서 매번 직접 검색·정렬·평가하는 시간이 들었다. 이미 신뢰할 만한 한국어 후기/가이드 글이 충분하니, 각 명소·맛집·활동에 대해 사전에 큐레이션된 블로그 포스트를 데이터에 박아두면 클릭 한 번에 본문으로 진입할 수 있다.

What

  • 스키마 확장: 명소 레벨 curated_links: [{title, url, source, snippet?}, ...], 맛집 link: {title, url, source}, 활동 string | {text, link?} 양쪽 허용 (역호환)
  • UI 렌더링: renderActivity / renderRestaurant / renderCuratedLink 헬퍼 3개. 큐레이션 블로그 카드 그리드 + 활동 inline 링크 + 맛집 카드 하단 링크. 기존 검색 그리드는 "더 찾아보기" 폴백으로 격하.
  • 데이터: 65/65 명소에 3개씩 curated_links (총 195개). 40/41 main 맛집에 1개씩 link. 48/172 활동 wrap.

How

Phase A — 스키마 + UI + Day1 main 3개 패턴 검증. Phase B — 7개 background subagent 병렬 dispatch (Day2 main, Day3 main, Day4–5 main, Day1 alt, Day2 alt, Day3 alt, Day4 alt). 각 subagent: WebSearch로 한국어 블로그 검색 → /tmp/curated-patches/<name>.json 저장 → 단일 merge 스크립트로 일괄 적용. 충돌 없음 (subagent는 read-only).

큐레이션 우선순위: (1) 한국 1차 블로그 (blog.naver.com, brunch.co.kr, *.tistory.com), (2) 한국 여행 플랫폼, (3) KKday/Klook 블로그, (4) 한국어 공식 가이드 (niche만).

Retrospective

잘된 점

  • 7개 background subagent 병렬 dispatch로 62 명소 큐레이션이 ~6분 wall-clock에 끝남. 단일 sequential이면 1시간+.
  • patch JSON 분리 + merge 스크립트 패턴 — subagent는 read-only, 메인만 places.json 라이터. 충돌·partial-write 없이 안전.
  • 스키마 옵셔널화로 Day1 main 3개 → 65개 확장 도중 기존 데이터/UI 회귀 0. "더 찾아보기" 폴백 보존.

다음에 가져갈 것

  • subagent가 가끔 출처가 모호한 사이트를 fallback으로 선택. "프로모션 가이드성 글보다 개인 후기 우선" 규칙을 prompt에 더 강하게 박을 것.
  • alt 명소의 활동도 link를 채우려면 별도 phase 필요. 현재 alt activities 124개는 의도적으로 plain string.
