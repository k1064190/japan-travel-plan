7단계

추상적 호텔 placeholder를 실제 예약 호텔 두 곳(수가타 신사이바시, 에비스히가시)으로 교체해 1~5일차 Google Maps 경로 출발지를 정확하게 수정.

Named Hotels

왜

이전까지 숙소는 namba_hotel 같은 추상 ID로 표현됐다. 실제 예약된 호텔 두 곳은 1~2박과 3~4박이 서로 다른 위치다. 데이터를 수정하지 않으면 Stage 6에서 복원한 Google Maps 경로 출발 좌표가 여전히 잘못된 곳을 가리켜 이동 시간 신뢰도가 다시 떨어진다.

무엇을

  • places.json에 3개 신규 항목 추가: sugata_hotel(수가타 호텔 오사카 신사이바시, Marriott Series), ebisuhigashi_hotel(나니와구 에비스히가시 2-1-24), ebisuhigashi_hotel_checkout(Day5 체크아웃 utility stop)
  • itinerary 호텔 참조 교체: Day1 체크인 → sugata_hotel; Day2 첫 stop transit origin + 마지막 stop → sugata_hotel; Day3 첫 stop origin + 체크인 stop → ebisuhigashi_hotel; Day4 → ebisuhigashi_hotel; Day5 첫 stop → ebisuhigashi_hotel_checkout
  • 도보 시간 보정: Day1/dotonbori 8→12분(수가타→도톤보리 ~900m, 기존 namba_hotel ~300m 기준 8분 부족)

어떻게

일회성 /tmp/patch-hotels.mjs 스크립트로 places.json과 itinerary.json 두 파일을 일관되게 한 번에 패치했다. 수가타 좌표는 사용자가 제공한 Google Maps URL의 정확한 값(34.6766637, 135.504813)을 사용. 에비스히가시 좌표는 주소 기반 추정(츠텐카쿠 동쪽 1~2블록, 34.6515, 135.508). 도보 시간 보정 후 validation과 verify-walk 모두 통과.

코드 위치

  • data/places.json — sugata_hotel, ebisuhigashi_hotel, ebisuhigashi_hotel_checkout 추가
  • data/itinerary.json — Day1~5 stops 및 transit_from_prev.origin_place_id 교체

회고

Stage 6에서 도입한 origin_place_id 필드가 정확히 이 시나리오를 위한 것이었다. 호텔 두 개를 데이터에 추가하는 것만으로 UI 코드 변경이 없었다. validate-data와 verify-walk 스크립트가 수가타→도톤보리 도보 시간이 짧다는 것을 즉시 플래그해 보정을 자동으로 발견하게 했다.
