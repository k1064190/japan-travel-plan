11단계

61개 식당 별점을 WebFetch로 직접 가져온 Tabelog 실측값으로 교체, 폐점 식당 1곳 처리 포함.

Real Ratings

왜

Stage 9에서 별점을 추가했지만 그 값은 subagent의 검색·추정치였다. UI에 (Tabelog)라고 출처 표기를 했음에도 실제 Tabelog 페이지를 직접 본 게 아니었다. Stage 10에서 별점 내림차순 정렬이 주요 탐색 기능이 됐으므로 별점의 정확성이 가장 큰 남은 과제였다.

무엇을

  • 61개 항목을 WebFetch 실측값으로 교체: alt 34개, 메인 명소 restaurants[] 27개
  • omen_kodaiji 폐점 처리: Tabelog 【閉店】 라벨 확인 → name_ko에 [폐점] 접두사, tags 첫 번째에 "폐점" 추가, rating 필드 제거(폐업 가게에 점수가 보이지 않도록)
  • 추정치와 실측의 주요 차이: katsukura_shijo 3.55→3.21, kamukura_dotonbori 3.52→3.20, mizuno_okonomiyaki 3.71→3.57, yamamoto_menzo 3.92→3.98, daiichi_asahi 후기 수 3,200→6,583

어떻게

4개 background subagent를 day area별로 병렬 dispatch. 각 subagent가 식당 목록을 받아 → 정확한 Tabelog URL 검색 → WebFetch로 페이지 fetch → 별점/후기 수 추출. 실패 19건(백화점 식품관·시장 노점·미등록 지점)은 failed[] 배열에 기록; 해당 항목은 추정치 대신 별점 미설정. 단일 /tmp/merge-ratings.mjs로 patch JSON 일괄 적용(alt는 ID, 메인 맛집은 이름으로 매칭).

코드 위치

  • data/places.json (61개 별점 갱신 + omen_kodaiji 폐점 처리)

회고

4개 병렬 subagent가 110+ 식당 WebFetch를 약 13분에 완료했다. Tabelog 폐점 라벨 검출은 추정치로는 절대 잡을 수 없었던 발견이었다. fetch 실패 19건은 Tabelog 커버리지 한계이지, 데이터 구조 문제가 아니다. 해당 항목을 별점 없이 두는 것이 올바른 선택이다.
