15단계

localStorage 영속성·우선순위 정렬·진행 progress bar를 갖춘 14개 예약 항목(유료 입장권 12개 + 라피트 교통편 2개) 추적용 예약 체크리스트 패널.

Reservation Checklist

왜

Stage 14까지 정보 표시·큐레이션·맵·UI는 완성. 출발 전 실제 사용자 액션은 예약 — USJ 입장권, 난카이 라피트, 오사카성·청수사·하루카스·가이유칸·도톤보리 리버크루즈 등은 사전 e-티켓이 가격·대기 모두 유리. 호텔은 이미 별도 예약 완료, 식당 웨이팅은 범위 밖. 별도 예약 체크리스트 패널을 통해 사용자가 진행 상태를 localStorage에 저장하며 추적한다.

무엇을

  • 스키마: place.booking (optional object) — required, url, advance_days, ticket_price_jpy, notes 필드 및 optional transit_from_prev.booking_url string
  • 12개 place에 booking 추가 — USJ, 오사카성, 하루카스300, 가이유칸, 톤보리 리버크루즈, 츠텐카쿠, 청수사, 뵤도인, 고다이지, 엔토쿠인, 기온코너, 후시미 사케 박물관
  • 2개 transit에 booking_url — Day 1 라피트 KIX→난바, Day 5 라피트 난바→KIX
  • renderBooking(place, color): 디테일 패널 Activities와 Restaurants 사이에 "예약" 섹션 삽입; 필수/선택 뱃지, 가격, 사전 권장일, 노트 표시
  • transit 카드에 인라인 "e-Ticket →" 버튼 추가
  • #checklist aside 패널 (데스크톱) + 모바일 3번째 탭: 14개 카드를 미완료 > 완료, 필수 > 선택, 사전 권장일 빠른 순으로 정렬; 완료 항목은 strikethrough + 회색 스타일
  • 진행 progress bar — "필수 N/M 완료"; 데스크톱 헤더 토글 버튼 "예약 X/Y"
  • localStorage["reservations"] JSON 객체로 영속; setReservationDone(id, bool) 헬퍼
  • 필수 8개 (USJ, 오사카성, 하루카스300, 가이유칸, 뵤도인, 기온코너 + 라피트 양방향); 선택 6개는 정보 표시용

어떻게

데이터 큐레이션을 subagent 4개 병렬로 처리 (Group 1: USJ + 라피트 KIX↔난바 via KKday; Group 2: 오사카 명소 5개; Group 3: 교토 명소 6개; Group 4: cross-check verifier). 각 subagent가 /tmp/booking-patches/에 patch JSON을 저장하면 /tmp/merge-bookings.mjs가 일괄 머지 (http→https 강제 변환 포함). Group 3에서 outdated 가격 정정: 뵤도인 600 ¥→700 ¥, 고다이지+엔토쿠인 통합권 900 ¥→1,200 ¥. validator는 validateBooking 헬퍼를 validateCuratedLink 패턴으로 추가; 새 테스트 케이스 4개 추가 (11→15 통과). collectReservationItems()가 places + itinerary 양쪽에서 booking을 수집한 뒤 정렬.

회고

4-subagent 병렬 큐레이션 패턴(Stage 11/12와 동일)으로 14개 항목을 약 16분만에 완성; Group 4 verifier가 cross-check 결과 0 corrections으로 마무리. 스키마는 옵셔널 필드만 추가하여 169개 place 전체 회귀 없이 통과. Group 3에서 plan 문서의 사전 가격이 outdated임을 확인 — 향후 데이터 작업 전 가격 정합성 사전 검증 단계를 계획에 포함할 것.
